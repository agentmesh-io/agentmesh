Project: AgentMesh

Context and assumptions
- `definition.txt` was empty; no Java sources were present. To move the project forward I made 2 reasonable assumptions:
  1) AgentMesh is a Java library/app that will manage a mesh of agents communicating over TCP/HTTP and exchanging JSON messages.
  2) The project should be small, runnable with Java 22, and include typical foundations: logging, JSON (serialization), networking (Netty), and a small DI/utility library.

Goal
- Provide a minimal but practical plan and the Maven dependencies needed to start implementing the AgentMesh core features.

Planned deliverables
1) A short-term MVP scope (0.5 - 2 weeks):
   - Core agent model (Agent id, state, message envelope)
   - Networking layer for agent-to-agent messaging (Netty-based TCP/HTTP transport)
   - JSON message (de)serialization with Jackson
   - Basic lifecycle (start/stop) and logging using SLF4J + Logback
   - Unit tests with JUnit Jupiter
2) Medium-term features:
   - Service discovery and mesh topology (gossip or simple registry)
   - Pluggable transport implementations
   - Observability (metrics and structured logs)

Technical contract (minimal)
- Inputs: JSON-serializable AgentMessages over TCP/HTTP
- Outputs: In-memory Agent instances, logs, and unit-testable behaviour
- Error modes: malformed messages (deserialize error), connection failure, invalid state transitions
- Success: agents can send/receive messages in unit tests with deterministic behaviour

Edge cases to consider
- Empty/invalid payloads
- Partial TCP frames (Netty handles framing but code should expect incomplete messages)
- High concurrency: thread-safety in agent state
- Long-running connections and reconnection handling

Required Maven dependencies (recommended minimal set)
- org.slf4j:slf4j-api (logging API)
- ch.qos.logback:logback-classic (logging implementation)
- com.fasterxml.jackson.core:jackson-databind (JSON serialization/deserialization)
- io.netty:netty-all (networking transport)
- com.google.guava:guava (utilities and immutable collections)
- org.junit.jupiter:junit-jupiter-api + engine (testing)

Suggested next development steps
1) Add a simple Agent POJO and AgentMessage DTO with Jackson annotations.
2) Add a tiny Netty server + client used by unit tests to exercise message send/receive.
3) Wire up SLF4J logging, add a sample configuration file under `src/main/resources/logback.xml`.
4) Implement unit tests that spin up in-memory transports to validate behaviour.
